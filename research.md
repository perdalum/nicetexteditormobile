# Research: Apple iPadOS/iOS frameworks for a plain text editor app in 2026

## Summary
Use a document-based app built around SwiftUI `DocumentGroup` + `FileDocument` for the simplest 2026 plain-text editor, but embed/wrap a UIKit `UITextView` for the editor surface rather than relying on SwiftUI `TextEditor` if you need serious iPad editing features. Apple’s standard stack gives you Files/iCloud Drive document access, in-place editing, text selection, find/replace, hardware keyboard shortcuts, sharing, and printing; line numbers, current-line highlight, syntax-like token highlighting, large-file tuning, shell/MPW worksheet execution, and go-to-line require custom work or are outside the standard/platform policy envelope.

## Findings
1. **Recommended document model: SwiftUI `DocumentGroup` + `FileDocument` for simple plain-text documents** — `DocumentGroup` declares the document types your SwiftUI app can open; its documents must conform to `FileDocument` or `ReferenceFileDocument`, and on iOS it adds a document browser that can navigate the file system plus multiwindow support. For common plain text, Apple notes you don’t need to declare OS-defined types such as `UTType.utf8PlainText`; do register `CFBundleDocumentTypes` for document-based opening. [Apple DocumentGroup](https://developer.apple.com/documentation/swiftui/documentgroup)

2. **Prefer UIKit `UITextView` as the editor surface for a real editor UI** — Apple’s TextKit guidance says framework text views (`UITextView`/`TextEditor`) give text input, selection, accessibility, undo/redo, dictation, inline predictions, and more “for free,” while custom TextKit views give more control but require implementing a production editing experience yourself. In a SwiftUI app, use `UIViewRepresentable` to host a `UITextView` when customization is needed. [WWDC26 TextKit](https://developer.apple.com/videos/play/wwdc2026/370/)

3. **TextKit 2 is the default modern text engine behind `UITextView`** — Since iOS 16, all UIKit text controls use TextKit 2 by default, including `UITextView`; Apple recommends transitioning to TextKit 2 and avoiding accidental fallback to TextKit 1. Do not access `UITextView.layoutManager` unless intentionally opting into TextKit 1, because Apple says that is the main cause of compatibility fallback and there is no automatic switch back to TextKit 2. [WWDC22 TextKit and text views](https://developer.apple.com/videos/play/wwdc2022/10090/)

4. **Files, iCloud Drive, and in-place editing are standard when using the document browser stack** — `UIDocumentBrowserViewController` browses local and cloud documents, can access the local and iCloud file providers by default, and can also work with third-party File Provider extensions. Apple explicitly warns not to assume files are local and to coordinate access using `UIDocument` or `NSFileCoordinator`/`NSFilePresenter` because files may live in iCloud Drive or other cloud providers and may change externally. [UIDocumentBrowserViewController](https://developer.apple.com/documentation/uikit/uidocumentbrowserviewcontroller)

5. **Open-in-place and Files visibility require the right Info.plist posture** — `LSSupportsOpeningDocumentsInPlace` indicates the app may open the original document from a file provider instead of a copy, and `UISupportsDocumentBrowser` identifies a document-based app. App Review guideline 2.5.15 also says apps that let users view/select files should include items from Files and the user’s iCloud documents. [LSSupportsOpeningDocumentsInPlace](https://developer.apple.com/documentation/bundleresources/information-property-list/lssupportsopeningdocumentsinplace), [App Review Guidelines 2.5.15](https://developer.apple.com/app-store/review/guidelines/)

6. **For document reliability, use `UIDocument`/coordinated file access if you go beyond basic `FileDocument`** — Apple’s document-based app guidance says `UIDocument` provides automatic file coordination, asynchronous reading/writing, change tracking, and conflict resolution. Recommendation: start with SwiftUI `FileDocument` for small/medium plain-text files; use `ReferenceFileDocument` or a UIKit `UIDocument` architecture if you need streaming, conflict handling, autosave control, or very large files. [Document-based apps](https://developer.apple.com/document-based-apps/)

7. **Find/replace is standard and should be enabled on `UITextView`** — `UIFindInteraction` provides a system find panel, result navigation, and optional replacement UI; hardware keyboard shortcuts such as Command-F, Command-G, and Command-Shift-G trigger it. `UITextView` has built-in support: set `textView.isFindInteractionEnabled = true`, then call `presentFindNavigator(showingReplace:)` when needed. [UIFindInteraction](https://developer.apple.com/documentation/uikit/uifindinteraction)

8. **Keyboard shortcuts are standard via `UIKeyCommand`/menus** — `UIKeyCommand` lets hardware-keyboard users invoke app commands, with standard commands such as Cut/Copy/Paste routed through responder methods and custom commands matched through the responder chain. Recommendation: add commands for New, Open, Save/Close if applicable, Find, Find Next/Previous, Go to Line, Share, Print, Toggle Line Numbers, and Escape/dismiss panels. [UIKeyCommand](https://developer.apple.com/documentation/uikit/uikeycommand)

9. **Sharing and printing are standard UIKit integrations** — Use `UIActivityViewController` with `NSItemProvider`/`UIActivityItemsConfiguration` to share UTF-8 plain text or a file URL; Apple’s sample uses `public.utf8-plain-text`. Use `UIPrintInteractionController` for AirPrint; it can print a ready item/file, formatter, or custom page renderer. [Sharing copies](https://developer.apple.com/documentation/uikit/collaborating-and-sharing-copies-of-your-data), [UIPrintInteractionController](https://developer.apple.com/documentation/uikit/uiprintinteractioncontroller)

10. **Text selection is best left to `UITextView`; custom text views are expensive** — TextKit 2 provides high-level selection/location abstractions and correct international text handling, while `UITextView` supplies the standard interaction affordances. A fully custom TextKit view can render and interact with text, but you give up system text view behavior and must rebuild input, selection, accessibility, edit menus, scrolling, and undo details. [Meet TextKit 2](https://developer.apple.com/videos/play/wwdc2021/10061/)

11. **Line numbers are possible but not a built-in editor switch** — Apple’s TextKit materials show line numbers as a customization layered around/subclassing text views and using viewport/layout-fragment information. For a 2026 app, implement a gutter synchronized with the text view’s scroll/viewport, cache line starts, and avoid full-document layout; treat wrapped visual lines vs logical newline-delimited lines as a product decision. [WWDC26 TextKit](https://developer.apple.com/videos/play/wwdc2026/370/)

12. **Current-line highlighting and syntax-like token highlighting are custom features** — `UITextView`/TextKit can display attributed text and custom rendering attributes/layout fragments, but Apple does not provide a plain-text-editor “current line highlight” or syntax highlighter. Recommendation: keep the persisted file as plain UTF-8; apply transient `NSAttributedString` attributes for token colors and current-line background, update incrementally around edits, and avoid TextKit 1 glyph APIs that can break complex-script correctness. [WWDC22 TextKit and text views](https://developer.apple.com/videos/play/wwdc2022/10090/)

13. **Large files need careful scope management** — TextKit 2 is designed around viewport-based, noncontiguous layout and Apple says this is essential for large documents, including documents hundreds of megabytes in size; however, `FileDocument` and `UITextView` usage commonly implies an in-memory string/attributed string. Recommendation: support a tested size tier for normal editing, warn/open read-only for very large files, and use `UIDocument`/streaming storage plus TextKit 2 viewport work only if large-file editing is a core requirement. [Meet TextKit 2](https://developer.apple.com/videos/play/wwdc2021/10061/)

14. **Go to line is not a system API; implement it yourself** — Use an incrementally maintained newline index to map line number → UTF-16/Swift string offset, then set `selectedRange` and call `scrollRangeToVisible(_:)` on `UITextView`. Do not compute line positions by laying out the whole document; for wrapped visual lines, use TextKit 2 `NSTextLayoutFragment`/`NSTextLineFragment` only near the viewport or after explicit expensive layout. [WWDC22 TextKit and text views](https://developer.apple.com/videos/play/wwdc2022/10090/)

15. **Shell/MPW worksheet execution is not a standard iPadOS feature and is risky for App Review** — App Review guideline 2.5.2 says apps must be self-contained and may not download, install, or execute code that introduces or changes app features/functionality; only educational apps have a limited exception for downloaded executable code, and the source must be completely viewable/editable by the user. Recommendation: do not ship arbitrary local shell/MPW worksheet execution in a plain text editor; if needed, provide non-executing worksheet markup, a built-in safe toy evaluator, or explicit remote execution on a user-controlled server with clear review notes and legal/security review. [App Review Guidelines 2.5.2](https://developer.apple.com/app-store/review/guidelines/)

## Gaps
- Apple does not publish a simple `UITextView` maximum editable file size; performance must be benchmarked on target iPad hardware with representative encodings, line lengths, and highlighting enabled.
- WWDC26 TextKit material includes newer customization examples; confirm exact SDK availability/deployment targets before depending on newly announced viewport hooks.
- App Review outcomes for any worksheet/interpreter feature are fact-specific. If execution is desired, get legal/App Review guidance early and document exactly what code can run, where it comes from, and whether it changes app functionality.
