NiceTextEditor for iPad and iPhone

A practical analysis of how to build a similar plain-text editor for iPadOS and iOS using standard Apple frameworks, preserving the good native editing ideas while intentionally dropping the MPW-like worksheet features and Go To Line.

1. Executive summary

Recommendation: build the iPad/iPhone app as a document-based SwiftUI app, but use a wrapped UIKit UITextView for the editing surface. This gives the most native result with the least custom editor code.

Keep

  • Plain-text document model.
  • Files/iCloud-based opening and saving.
  • Native text selection, undo/redo, spelling, keyboard input, accessibility.
  • Find/search using Apple’s standard find UI.
  • Text size and simple appearance controls.
  • Optional visual-only .VB/.VE and .QB/.QE markup.
  • Optional iPad-first line numbers if they are kept simple.

Do not port

  • MPW/BBEdit-like UNIX worksheet.
  • Persistent per-document /bin/zsh process.
  • Pipeline filtering of selected text.
  • WorksheetStartup.zsh and shell shortcut settings.
  • Go To Line, as requested.
  • macOS-specific menus, Font panel, AppKit ruler view, AppKit alerts.

The best iOS version is not a mechanical port. It should be a smaller native mobile document editor that borrows the current app’s philosophy: ordinary files, no proprietary state, a comfortable editor surface, and a few tasteful visual conveniences.

Core features found in the macOS app

  • Plain-text open/save with standard document behavior.
  • Proportional system font by default, configurable on macOS via Font panel.
  • Per-document text size, tab width, foreground/background colors.
  • Line-number gutter.
  • Standard text assistance: spelling, grammar, correction, replacements, smart punctuation, link/data detection.

3. Recommended iPadOS/iOS platform model

Use standard Apple kits: SwiftUI for app/document chrome, UIKit for the editable text view, Foundation/UniformTypeIdentifiers for file representation, and system document integration for Files/iCloud.

Document handling

Use SwiftUI DocumentGroup with a mobile version of PlainTextDocument conforming to FileDocument. On iPadOS/iOS this gives the standard document browser experience and integrates naturally with Files, iCloud Drive, and other file providers.

Recommended document posture:

  • Register as an editor for public.plain-text, public.text, and probably public.utf8-plain-text.
  • The macOS code also accepts .sourceCode and .data at the FileDocument level. For mobile, preserve .sourceCode if you want scripts/config files to appear naturally in the document browser, but be cautious with broad .data: it can expose arbitrary binary files that will usually fail decoding or produce a poor user experience.
  • Set LSSupportsOpeningDocumentsInPlace so the app edits original files from Files providers when allowed.
  • Set UISupportsDocumentBrowser if using a UIKit document-browser-based shell; SwiftUI DocumentGroup handles much of this for a SwiftUI document app.
  • Read UTF-8 first; optionally fall back to UTF-16 and MacRoman as the macOS app does.
  • Write UTF-8. Keep files plain.

Text editor surface

Use UITextView wrapped in UIViewRepresentable. SwiftUI’s built-in TextEditor is convenient, but it is too limited for line numbers, find integration control, text-container insets, attributed visual markup, tab width, keyboard commands, and robust editor behavior.

UITextView provides:

  • Native selection, insertion point, edit menu, drag/drop, copy/paste.
  • Undo/redo integration.
  • TextKit 2-backed layout on modern systems.
  • Spelling, autocorrection, smart punctuation and keyboard behavior.
  • Accessibility, dictation, hardware keyboard interaction, pointer support.
  • UIFindInteraction support for find/replace.

iPad and iPhone adaptation

The app should be one codebase with size-class-aware UI:

  • iPad: richer toolbar, optional line numbers, multiwindow support, keyboard shortcuts, Stage Manager/split-view safe layout.
  • iPhone: minimal toolbar, document title + count, find/share/appearance controls, line numbers off or hidden.

5. Features that are unsuitable, hard, or not worth porting

MPW-like worksheet / shell execution

The current macOS app’s worksheet is built around Process, Pipe, a persistent /bin/zsh, and a user-managed startup file. That is natural on macOS, but not a standard app pattern on iPadOS/iOS.

Reasons not to implement it:

  • iPadOS/iOS apps are sandboxed and do not have the same general-purpose shell environment.
  • Arbitrary command execution is a security and policy risk.
  • It would require a non-standard interpreter, remote execution service, or a heavily constrained toy environment.
  • It would dominate the project’s complexity while not being required for a personal mobile text editor.

Conclusion: exclude it entirely. Keep visual markup if desired, but no execution.

Line numbers

The macOS implementation uses NSRulerView, NSTextView.layoutManager, visible glyph ranges, and logical newline starts. UIKit has no drop-in NSRulerView equivalent. A mobile gutter must be custom and synchronized with scrolling, content insets, wrapping, dynamic type/font changes, and keyboard movement.

Conclusion: possible, but not a first-version feature. If implemented, make it iPad-first, logical-line-only, and optional.

Live visual markup on large files

The macOS app restyles the text storage after edits. The same idea is possible with UITextView.attributedText, but full-document rescans and full attributed-string replacement can be expensive and may disturb selection if done carelessly.

Conclusion: implement with debouncing and test with realistic files. Disable or simplify for large documents.

Large-file editing

The existing app’s model loads documents into a String. That is appropriate for normal notes, scripts, README files, and prose, but not for huge logs. A serious large-file editor would need streaming storage, viewport-based layout strategy, and much more testing.

Conclusion: set a tested size policy. Warn or open read-only for large files rather than pretending this architecture is a log viewer.

Go To Line

You explicitly requested that the iPadOS/iOS version not include Go To Line. It is also a reasonable omission: system find is more valuable on mobile, and Go To Line adds UI and indexing complexity for little personal benefit.

6. Recommended architecture

High-level module structure

  PlainTextDocument.swift            // FileDocument text load/save
  DocumentEditorView.swift           // SwiftUI document screen and toolbar
  TextEditorView.swift               // UIViewRepresentable wrapping UITextView
  TextStyler.swift                   // optional .VB/.QB visual-only styling
  EditorTheme.swift                  // UIColor/Color palette
  EditorSettings.swift               // UserDefaults-backed preferences
  ShareAndPrint.swift                // UIActivityViewController / UIPrintInteractionController bridges

Core document type

The existing PlainTextDocument can be adapted almost directly. Keep it value-type and simple first:

struct PlainTextDocument: FileDocument {
    static var readableContentTypes: [UTType] = [.plainText, .text, .utf8PlainText]
    static var writableContentTypes: [UTType] = [.plainText, .utf8PlainText]

    var text: String = ""

    init(configuration: ReadConfiguration) throws {
        guard let data = configuration.file.regularFileContents else {
            throw CocoaError(.fileReadCorruptFile)
        }
        if let utf8 = String(data: data, encoding: .utf8) {
            text = utf8
        } else if let utf16 = String(data: data, encoding: .utf16) {
            text = utf16
        } else if let macRoman = String(data: data, encoding: .macOSRoman) {
            text = macRoman
        } else {
            throw CocoaError(.fileReadInapplicableStringEncoding)
        }
    }

    func fileWrapper(configuration: WriteConfiguration) throws -> FileWrapper {
        FileWrapper(regularFileWithContents: Data(text.utf8))
    }
}

Editor view strategy

Create a UIViewRepresentable that owns/configures UITextView. The coordinator should bridge changes back to the SwiftUI document binding, preserve selection when applying programmatic changes, and expose methods for find, zoom, and optional styling.

struct MobileTextView: UIViewRepresentable {
    @Binding var text: String
    var fontSize: CGFloat
    var theme: EditorTheme
    var showMarkup: Bool

    func makeUIView(context: Context) -> UITextView {
        let view = UITextView()
        view.delegate = context.coordinator
        view.isEditable = true
        view.isSelectable = true
        view.alwaysBounceVertical = true
        view.keyboardDismissMode = .interactive
        view.isFindInteractionEnabled = true
        view.textContainerInset = UIEdgeInsets(top: 14, left: 14, bottom: 160, right: 14)
        return view
    }

    func updateUIView(_ view: UITextView, context: Context) {
        // Apply text, font, theme, insets, and optional attributed markup carefully.
    }
}

State model

  • Document state: the text only.
  • Per-scene/editor state: font size, theme, tab width, line-number visibility.
  • Global user preferences: default font size/theme, smart-punctuation preference, default tab width.
  • Not stored in file: colors, zoom, line numbers, UI state, markup rendering state.

7. Suggested implementation plan

Version 0.1 — private usable editor

  1. Create a new iOS/iPadOS SwiftUI document app target.
  2. Port PlainTextDocument and Info.plist document type declarations.
  3. Build DocumentEditorView with a UITextView wrapper.
  4. Support open, edit, autosave/save, and new document flows through the system document UI.
  5. Add text size controls and a small theme picker.
  6. Add character count and document name.
  7. Enable system find.
  8. Add share sheet for the current file/text.
  9. Test with hardware keyboard, software keyboard, iPad split view, Stage Manager, and iPhone portrait.

Version 0.2 — NiceTextEditor personality

  1. Add visual-only .VB/.VE monospace blocks.
  2. Add visual-only .QB/.QE italic quote blocks.
  3. Add tab width rendering if it behaves well with attributed text.
  4. Add print support.
  5. Add keyboard shortcuts for find, zoom, share, and maybe appearance.

Version 0.3 — optional editor refinements

  1. Prototype iPad-only line-number gutter.
  2. Add large-file warnings and possibly read-only mode.
  3. Improve multiwindow behavior on iPad.
  4. Add a compact settings screen.

8. Risks and experiments

The analysis used repository inspection and Apple-framework research. Before committing to the final implementation, run these small experiments in Xcode on the target devices:

Important risk notes

  • TextKit versioning: avoid relying on old TextKit 1 glyph APIs unless there is a deliberate fallback strategy.
  • Attributed text churn: replacing all attributed text on each keystroke can reset selection or hurt performance.
  • File providers: Files documents may be remote or externally changed; standard document APIs help, but testing with iCloud Drive matters.
  • Private use does not remove platform constraints: even for your own app, arbitrary shell execution is still a poor iOS fit.

These are the main Apple frameworks and APIs that should be consulted while implementing the design:

  • SwiftUI FileDocument for serializing plain-text documents.
  • UniformTypeIdentifiers UTType for document type declarations.

  • UITextView and TextKit for the editing surface.
  • UIFindInteraction for system find and replace UI.

  • UIActivityViewController for sharing.
  • UIPrintInteractionController for AirPrint.

10. Conclusions and recommendations

  1. Build a new iPadOS/iOS app, not a direct port. The existing repository is a good design reference, but much of its code is AppKit-specific.
  2. Use SwiftUI DocumentGroup and FileDocument. This preserves the current app’s plain-document model and gives native Files/iCloud behavior.
  3. Use UIKit UITextView, not a custom editor. This is the most reliable path to native editing, selection, undo, keyboard, accessibility, and find.
  4. Exclude the MPW worksheet and Go To Line. The worksheet is unsuitable for iPadOS/iOS; Go To Line is explicitly out of scope and not worth the mobile UI complexity.
  5. Preserve the quiet strengths: plain text, tasteful themes, zoom, tab-width control, visual-only markup, character count, comfortable bottom scrolling.
  6. Defer line numbers. They are possible, but they are a custom UIKit/TextKit feature and should not block the first usable version.
  7. Set realistic file-size expectations. An in-memory text editor is fine for normal personal documents, not huge logs.
  8. Keep the app boringly private. No analytics, no network layer, no third-party frameworks unless necessary, no arbitrary code execution.
Final recommendation: first build a small native document editor with DocumentGroup + UITextView. Once editing and Files behavior are solid, add NiceTextEditor’s personality through simple themes and visual-only markup. Consider line numbers only after that, and keep them iPad-first.
